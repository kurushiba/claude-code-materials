基本構造と作り方

  • .mdファイルとして定義（ケバブケース名）
    例: code-reviewer.md
  • ファイルの中身は2層構造
    上部: YAML（設定）、下部: Markdown（指示）
  • /agents コマンドで自動生成がおすすめ！
    対話形式でひな形を作ってから微調整
サブエージェントファイルの構造
---
name: qa-tester
description: テストコードの作成時に使用
tools: Read, Grep
---

あなたは優秀なQAテスターです。
以下のルールに従ってテストを作成...

保存場所（スコープ）と優先順位

  • 置く場所で「使える範囲」が変わる
  • 一番よく使うのはプロジェクトスコープ
    (Gitでチーム共有)
  • 同名の場合は優先度順で1つだけ適用
優先順位（高い順）
1
管理スコープ
全社員向け (管理ディレクトリ .claude/agents/)
2
プロジェクトスコープ
チーム共有 (プロジェクト直下 .claude/agents/)
3
ユーザースコープ
個人横断用 (~/.claude/agents/)
4
プラグインスコープ
プラグイン同梱

YAML設定の主要項目

name / description

必須項目。
特にdescriptionは「いつ呼ぶか」の
自動委譲の判断基準になる重要項目。

tools / disallowedTools

権限の制御。
読み取り専用にするなど、
最小特権の原則を実現する。

model

使うモデルを指定。
(sonnet / opus / haiku / inherit)
軽い処理はhaiku、重要な実装はopus、
といった使い分けが可能。

サブエージェントの呼び出し方

  • 自動委譲
    Claudeがdescriptionを読んで自律的に判断
  • 自然言語での明示
    「コードレビュー用エージェントを使って」と指示
  • @メンション（一番確実）
    プロンプト内で「@agent-code-reviewer」と指名
  • 起動時コマンド
    claude --agent <エージェント名> で専用セッション
確実な呼び出しイメージ
ユーザー入力:
このファイルのレビューをお願いします。
@agent-code-reviewer
Code Reviewer エージェントが起動！

サブエージェント作成のTips

  • 1. トリガーを明示する（PROACTIVELY）
    descriptionに「〜の時は積極的に使う」とアクション志向で書く。
  • 2. ツール権限の最小化
    破壊的変更を防ぐため、レビュー用ならRead, Grep等のみ許可する。
  • 3. 役割を詰め込みすぎない
    「単一の目的に特化した専門家」として設計する。
  • 4. ネスト（入れ子）はできない
    エージェントからエージェントは呼べない。Agent Teamsなどを活用する。
