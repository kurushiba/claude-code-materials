Skillsディレクトリ構成の基本ルール

  • 1つのコマンド ＝ 1つのディレクトリ
  • ディレクトリ名がコマンド名になる
  • 命名ルールはすべて小文字の「ケバブケース」
    （例：my-custom-skill）
  • 注意: 直下にREADME.mdは置かない
📁 my-custom-skill/← コマンド名: /my-custom-skill
├── 📄 SKILL.md（必須）
├── 📁 scripts/
├── 📁 references/
└── 📁 assets/

Skillフォルダの4つの構成要素

  • SKILL.md (必須)
    Skillの心臓部。YAML設定と指示プロンプト。
  • scripts/ (任意)
    実行するPythonやBashスクリプトを格納。
  • references/ (任意)
    APIリファレンス等の参照用ドキュメント。
  • assets/ (任意)
    テンプレートや雛形などのアセットファイル。

YAMLフロントマターで挙動制御

  • SKILL.mdの上部に --- で囲んで記述
  • Skillの挙動を細かく制御するための設定
  • description が最も重要！
    （Claudeが自動発動の判断に使用する）
---
name: my-skill
description: このSkillの説明
disable-model-invocation: false
allowed-tools: Read, Bash(git:*)
---
項目名 役割・説明
name Skillの識別子（任意）
description 「いつ・何をするか」の説明（最重要）
disable-model-invocation trueでClaudeによる自動発動を禁止
allowed-tools ユーザー承認なしで実行を許可するツール

コマンド引数の受け取り（$ARGUMENTS）

  • ユーザーが入力したテキストをそのまま埋め込む
  • 未記述の場合は自動でプロンプト末尾に追加される
  • $0・$1 でスペース区切りの個別取得も可能
$ /deploy staging
↓
プロンプト内の $ARGUMENTS が...
「staging」に置き換わる！

絶対パスとセッションID

  • ${CLAUDE_SKILL_DIR}
    Skillディレクトリの絶対パス。同梱スクリプトの呼び出しに必須。
  • ${CLAUDE_SESSION_ID}
    現在のセッションUUID。ログファイル名の一意化などに活用。
# どこから実行しても確実にスクリプトを呼び出せる
bash ${CLAUDE_SKILL_DIR}/scripts/validate.sh

ファイル内容の参照（@シンボル）

  • @ を使うとファイルの内容を読み込める
  • Claudeが指示を実行する前に、自動で中身を展開
  • コードの比較や、特定ファイルへの処理指示に便利

  • プロンプトを読む直前にバックグラウンドで実行
  • 実行結果（標準出力）がそのままテキストとして埋め込まれる
  • 最新のプロジェクト状態（git diff等）を渡すのに最適
