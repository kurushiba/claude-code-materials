Skillsの2つの役割

Skillsは、Claudeに特定のタスクやワークフローを教える「再利用可能な指示セット」です。

  • 参照用スキル（知識）
    APIのコーディング規約やデザインガイドライン。「このルールに従って」という判断材料。
  • タスク用スキル（行動）
    デプロイ手順やバグ修正の流れ。「この順番で操作して」という作業のレシピ。
  • Agent Skills規格
    オープンスタンダードを採用。将来は他のAIツールでも使えるポータビリティ。
参照用スキル 📖
「知識・ルール」を教える
例：コーディング規約、アーキテクチャ方針
タスク用スキル 🛠️
「行動・手順」を教える
例：デプロイ手順、PR作成のステップ

CLAUDE.mdとの違い

どちらもClaudeに指示を出すものですが、読み込まれるタイミングと目的が違います。

CLAUDE.md Skills
プロジェクト全体のルール
常時読み込み
  • すべてのセッションで常にコンテキストに入っている
  • プロジェクトの基本的なDNA、絶対的な約束事を書く
オンデマンドのプレイブック
必要な時だけ呼び出し
  • 必要な状況でのみ読み込まれる
  • 特定のタスク専用の手順や、一部の領域だけで使う深い知識を書く

Skillsの基本構造

Skillは単一のファイルではなく、ディレクトリ（フォルダ）として構成されます。

  • SKILL.md が中心
    フォルダの中心には必ず SKILL.md があります。
  • メタデータと指示の本体
    冒頭のYAMLに「Skillの説明」などのメタデータを書き、その下に指示を書きます。
  • 補助ファイルも同梱可能
    仕様書やスクリプトなどのアセットを同じフォルダに置いて連携できます。
📁 my-deploy-skill/ （Skillフォルダ）
📄 SKILL.md ← 中心の指示ファイル
📁 scripts/ ← 実行可能なコード
📁 references/ ← リファレンス等
📁 assets/ ← テンプレートや雛形

段階的開示（Progressive Disclosure）

Skillをたくさん登録してもコンテキストを圧迫しないよう、3段階で少しずつ読み込みます。

第1段階：セッション開始時
メタデータのみ

名前と短い説明文だけをインデックスとして読み込む。

📚 例え： 本棚に並んだ本の「背表紙のタイトル」だけを見ている状態

▶
第2段階：必要になった時
SKILL.md本体

質問と一致するか手動で呼ばれた時に、具体的な指示を展開。

📖 例え： 料理を作る時に、必要な料理本を「1冊だけ手に取って開く」状態

▶
第3段階：さらに詳細が必要な時
補助ファイル

タスクを進める中で、Claudeが必要と判断したら補助ファイルを読みに行く。

📝 例え： 細かい別冊レシピなどを「その時だけ」取り出す状態

プラグインとマーケットプレイス

作ったSkillやカスタマイズ環境を、チームや組織に配布・共有する仕組みです。

  • プラグイン
    Skills、Hooks、MCP設定などを1つにまとめたパッケージ。他のリポジトリでも即座に同じ環境を再現できます。
  • 名前空間で衝突防止
    プラグイン名:スキル名 の形式になり、同じ名前のSkillがあっても干渉しません。
  • マーケットプレイス
    プラグインを配布する場所。Gitリポジトリやnpmパッケージなどからインポートします。

外部プラグイン利用時の注意点

⚠️ 必ず中身を確認してから使う！

プラグインは、あなたの開発環境の中でClaude Codeの動作に直接影響を与えます。想定外の自動操作（ファイル削除など）が走るリスクもあります。

1

信頼できる提供元か確認する

公式のプラグインか、信頼できる開発者が提供しているものかを確認する。

2

SkillsやHooksの中身を読む

インストール前に、どんな指示が書かれているか、どんな操作をするのかを自分で確認する。

3

安全と判断してからインポート

内容に問題がないことを確認した上で、自分のプロジェクトに適用する。
